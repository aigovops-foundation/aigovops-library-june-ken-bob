The Library — Durability Plan

How the system stays up, what makes it sleep, and the proper end-to-end shape.

The Library lives in two layers. The hub (this site, on GitHub Pages) is served by a CDN and is effectively always-up. The core (the API, the agents, the Beacon-signed ledger) runs on Fly.io and is the layer that's behaved like it "goes down." It hasn't been crashing — it's been sleeping on idle, by default. This plan ends that and lays out the durable shape we want next.

1 · The end-to-end design today

Static layer — GitHub Pages. The hub (index, tour, rooms, design book, plan, blueprint, pulse, the live Gates demo, the recorded walkthrough) is plain HTML/JS/MP4 served by GitHub's CDN. No machine to sleep, no volume to lose. Currency comes from a build/test gate on every push.

Dynamic layer — Fly.io. One app (aigovops-library-core) on one machine, one region (iad), one volume, with scale-to-zero on. Surfaces: /, /console, /tour, /demo, /newsroom, /status, /auth/login, /api/socrates. Identity is GitHub OAuth (real). Members propose; stewards (Bob, Ken) hold the gate. Every accepted decision becomes a Beacon-signed receipt in the ledger (NDJSON on the volume).

The honest single points of failure, in order:

  1. Auto-stop (certain to bite — it's the default behavior): the machine suspends on idle and cold-starts on the next request, which reads as "down" for ~1–3 seconds.
  2. Single machine, single region: a host issue takes the URL down until Fly reschedules.
  3. The ledger volume: the signed history lives on one disk. For a governance product, this is the crown jewel — if it goes, the system of record goes.

One thing the design already does well: the live Gates demo degrades gracefully — when the core is unreachable it shows an honest OFFLINE badge with embedded data instead of breaking. Keep that.

2 · The durability tiers

Tier 1 — Never sleeps (config only, ~5 minutes)

Disable scale-to-zero and pin one machine running. In fly.toml's [http_service] block:

auto_stop_machines = false

auto_start_machines = true

min_machines_running = 1

Then:

~/.fly/bin/flyctl scale count 1 --app aigovops-library-core

~/.fly/bin/flyctl deploy --app aigovops-library-core

Cost: ~a few dollars/month for one always-on shared-cpu-1x. Eliminates "looks down."

Tier 2 — Survives a single machine failure

Run two machines behind Fly's proxy with an HTTP health check on /status:

~/.fly/bin/flyctl scale count 2 --app aigovops-library-core

The catch — and it's real: Beacon keys and the NDJSON ledger live on a per-machine volume. Two machines means two volumes that don't share state; receipts would split. So Tier 2 is only correct after Tier 3.

Tier 3 — State that survives anything (the real shape)

Move shared state off any single machine:

  • Ledger → managed Postgres. The code already supports it: createStore() falls back to FileStore only when DATABASE_URL is unset. So this is config, not new code. Provision a small managed Postgres, set DATABASE_URL as a Fly secret, redeploy.
  • Beacon keys → Fly secrets (read at boot, never on disk).
  • Daily ledger export → private repo (a small scheduled job exports NDJSON snapshots and commits them; survives even total infra loss; matches the product's promise of durable, tamper-evident records).

When Tier 3 is in place, Tier 2 (scale count 2) becomes safe and correct.

3 · Operational hygiene

  • Health check. Add a /status HTTP check to [http_service.checks] so Fly restarts a hung machine without manual intervention.
  • Uptime monitor. A free external pinger (UptimeRobot or similar) on the hub URL and /status, alerting Bob on first failure. Cheapest, highest signal.
  • Backup keys. A second Beacon keypair stored offline (printed or in 1Password), so a key-rotation can be done without a key-recovery drama.
  • The canonical URL we share is the hub. The hub is the always-up front door. The core is the engine behind it. The live Gates demo already embodies this: it loads from the CDN and lights up LIVE only when the core answers.

4 · Honest recommendation, in order

  1. Today — Tier 1. Disable auto-stop. One always-on machine. This ends the visible problem and costs a couple of dollars a month. Five-minute change.
  2. This week — Tier 3a. Provision managed Postgres, set DATABASE_URL, redeploy. The code already speaks Postgres; this is the durable shape.
  3. When ready — Tier 2. Scale to two machines behind the proxy with health checks. Safe to do after Postgres.
  4. This week, parallel — daily ledger export to the private repo. Cheapest insurance for the system of record.
  5. Always — uptime monitor + backup keys. Hygiene.

What we are not doing, deliberately: we are not multi-region, multi-cloud, or running a GPU model in production yet. Those are later decisions, made with evidence — exactly per the gate law. We hold them as Maybes with owners, not chase them as Yeses without.
