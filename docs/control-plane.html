00 The promise

Five commitments. Everything below exists to make these true in code, not in policy prose.
"Agents do the bureaucracy; humans hold the meaning — and humans hold the keys."
  • One artifact, three homes. The exact same container runs on a lab laptop, on the public community cloud, and inside a customer's firewall. Only a profile changes — never the code.
  • No agent acts except through the gate. An agent has no ambient credentials, no shell, no network. Its only way to affect the world is a typed proposal to the policy gateway. That single constraint is what makes "gone wild" structurally impossible.
  • Propose, don't execute. Anything irreversible pauses for a human, with a one-line plain-language summary. Capability is a reversible dial that widens with trust, never a blank cheque.
  • We broker secrets; we never hoard them. No password ever lives in this platform or in an agent. A vault holds them; the gate hands out short-lived, narrowly-scoped tokens that expire on their own.
  • The receipt is the unit of truth. Every meaningful effect emits an append-only, Ed25519-signed record. Tail the ledger and you have total real-time visibility of effects — plus a break-glass switch to halt instantly.

★ Decisions · Rev 2026.06

Settled calls that the sections below assume. The principle: define the safety contract once at the interface; enforce it with the strongest backend each environment allows; scope every view by identity.
  • Secrets: tiered, behind one interface. A single SecretsProvider contract — getToken(scope, ttl) · rotate() · revoke() — with two adapters: a FileProvider (lab: OS keychain / .env) and a VaultProvider (community + enclave: HashiCorp Vault or cloud KMS). The gateway only ever calls the interface, so there is one call site. The broker semantics are identical everywhere — even the lab adapter mints a short-lived, scoped, revocable, logged token and never hands an agent a raw secret. Documented asymmetry: Vault issues truly dynamic ephemeral creds in prod; the lab simulates short-lived with an enforced-TTL wrapper over a static secret (acceptable because the lab touches no real targets).
  • Chokepoint: sandbox contract from day one; gVisor where the kernel allows. The contract is enforced everywhere from the start — tools run sandboxed, no ambient network or filesystem, egress only through a declared proxy. The backend is gVisor (kernel-enforced isolation) on Linux community/enclave hosts, with a documented lighter fallback on dev laptops (seccomp default profile + network-namespace isolation + read-only rootfs). The guarantee — an agent can never open a socket or touch the host — holds in every home; gVisor only makes it stronger where it can run.
  • Oversight: one surface, role-scoped. Same code for stewards and members. Stewards (the founders) see every receipt across all agents, the capability dials, secret-issuance events, and the armed global kill switch. A member sees only their own effects and can pause only their own workflows — never others' data, never the global switch. Mechanically: every receipt is tagged owner + scope; the console is a live (SSE) filtered view over the ledger gated by the viewer's authz; pulling the global kill switch is itself a steward-only privileged action that emits its own signed receipt.
Three contracts — SecretsProvider, the sandbox boundary, and the oversight view — same semantics everywhere; only the backend and the visibility vary, by environment and by role. That is what keeps the guarantee from weakening just because you changed homes or who's looking.

01 One artifact, three homes

Write once. Refuse to fork it for any environment. The PROFILE selects everything else.

The cleanest way to run "local for a lab / on the website for the community / inside a firewall / in a container" is to ship a single image and a single codebase, with one PROFILE env var that selects the datastore, models, secret store, egress policy, and capability ceiling. Same binary everywhere; only configuration differs. That is the anti-lock-in guarantee made literal — a regulated org can run the whole thing air-gapped and verify our published ledgers offline with openssl.

02 The single chokepoint

Don't watch the agents. Make the unsafe path impossible.
FIG 01 — the chokepoint: the dashed red path does not exist. Injection, a haywire model, or a logic bug can only emit a proposal the gate rejects.

The instinct is to watch the agents. The stronger move is to make the unsafe path impossible rather than monitored. An agent has no ambient anything — no shell, no network, no credentials, no filesystem beyond a scratch space. Its only way to affect the world is to emit a typed proposal to the gate. If the agent's prompt gets injected, its model goes haywire, or its logic has a bug, the worst it can do is emit a proposal the gate rejects. "Gone wild" stops being possible because there is no wire from the agent to the world that bypasses the gate.

03 Agent safety — bullet-proof by construction

Determinism where we can, hard limits everywhere, and a switch that always wins.
  • The Yes-Gate is pure functions. Input → decision, no side effects. The same evaluation runs client-side for preview and server-side for the record, so it is exhaustively unit-testable and identical wherever it runs.
  • Policy as written code. Gates compile to Open Policy Agent (OPA) — open-source, embeddable, runs local or in-cluster. "Every action evaluated against written policy" comes for free, and policy changes are reviewable in pull requests.
  • Capability dial + hard caps. Every member and agent has an explicit, reversible capability level and hard caps on spend, request rate, and blast radius. The agent pauses at the cap rather than pushing through. Defaults are narrow; trust widens them; one toggle narrows them again.
  • Sandboxed tools, least privilege. Agent tools run in isolated contexts with a declared egress allow-list — no ambient credentials, no network beyond the endpoints a tool explicitly names.
  • Kill switch & circuit breakers. Any deployed workflow can be paused instantly. Because effects are signed and reversible-by-design, recovery is "Recover to Yes," not a forensic scramble.
  • Build-your-own-agent, governed. Members compose agents in the Makerspace; those agents inherit the same gates, caps, and signing — capability is a dial they earn, not a switch they flip.

04 The credential & secrets control plane

Total visibility of every account — without the platform ever holding a plaintext secret.
The firm rule. We do not build a password store, and agents never hold credentials. A homegrown vault, a secrets spreadsheet, or passwords in the repo would be exactly the failure a governance Foundation exists to prevent. We keep an inventory of every account; we never take custody of the plaintext.
FIG 02 — the secrets broker: agents get scoped, minutes-long tokens; the master credential stays in the vault; the registry gives full visibility without custody.
  • Humans' credentials live in a real manager (1Password / Bitwarden) with MFA and a shared team vault. That is the inventory's source of truth for human logins.
  • Machine / agent secrets live in HashiCorp Vault — open-source, self-hostable, identical across lab, cloud, and enclave. When a proposal is approved, the gate requests a short-lived, narrowly-scoped token (minutes, one purpose), hands it to the sandboxed tool, and it expires by itself.
  • Config in git is encrypted with SOPS + age — secrets are never committed in plaintext but stay version-controlled and reviewable.
  • The account registry records every account: owner, scope, last-rotated, and which agent (if any) may request a token against it. Full visibility; zero plaintext custody.
  • Rotation & revocation are first-class: rotate at the vault, and every short-lived token derived from it ages out on its own. Revoke a capability and the next request simply fails closed.

05 Real-time oversight — what "100%" honestly means

Total visibility of effects, instant control, and an honest line about what's knowable.

Be precise so the promise is true: you can have complete visibility of every effect — not 100% prediction of a model's thoughts, but 100% of what it actually does — because every action flows through the one gate and emits a signed receipt to the append-only ledger. Tail that ledger over server-sent events and you get a live oversight console: every proposal, every gate decision, every token issuance, every cap state, in real time. "Control" means a break-glass kill switch plus caps that stop before damage. That is the honest version of "100%": total observability of effects, and the power to halt instantly.

The oversight spectrum — "no agents without oversight" sits on a dial, and the capability level chooses where each action lands:

  • In-the-loop — a human approves each action. The default. Required for anything irreversible.
  • On-the-loop — a human monitors a live feed and can interrupt. For reversible, capped, well-tested actions that have earned trust.
  • Over-the-loop — audit after the fact via the signed ledger. Only for low-risk, fully-reversible, rate-capped work.
Start paranoid; loosen deliberately, per capability, never globally — and the kill switch is always live.

The tradeoff stated plainly: full in-the-loop maximizes safety but minimizes the labor agents save. The capability dial is the resolution — earn autonomy one specific, reversible, capped action at a time, with on-the-loop monitoring and an instant halt behind it.

07 Honest boundaries — what we won't do

A governance project is judged by the lines it refuses to cross, including for itself.
  • No plaintext secret custody. The platform never stores a password or long-lived key. Vault holds them; we broker scoped, expiring tokens.
  • No agent-held credentials. An agent never receives a master secret — only a minutes-long token for one declared purpose.
  • No autonomous irreversible action. Destructive or external-effect operations are quarantined into proposals behind a human gate, enforced in the runtime — not left to prompt discipline.
  • The irreversibility boundary holds for the builders too. Tooling (including AI assistants) prepares configs, values, and verification; a human makes the irreversible clicks — DNS, registrar, key enrollment, account creation, deletion — after the risk is flagged.
  • "100%" is about effects, not minds. We promise total visibility and control of what agents do, and we say so plainly rather than overclaiming prediction of what they might think.
